User Management for New Team Members

1

📋 Scenario

Your company just hired 5 new developers. You need to create user accounts, set up their home directories, assign them to the "developers" group, and ensure they change their password on first login.

Requirements:
  • Create users: alice, bob, charlie, diana, eve
  • All users must belong to "developers" group
  • Home directories: /home/username
  • Force password change on first login
  • Set password expiry to 90 days
useradd usermod passwd chage
Group Creation:
First create the developers group: sudo groupadd developers
Verify it was created: getent group developers
User Creation Command:
Create user with specific group: sudo useradd -m -g developers -s /bin/bash alice
• -m: Create home directory
• -g developers: Primary group
• -s /bin/bash: Default shell
Repeat for all users or use a for loop
Password Configuration:
Set initial password: sudo passwd alice
Force change on first login: sudo chage -d 0 alice
Set 90-day expiry: sudo chage -M 90 alice
View password status: sudo chage -l alice
Script for Multiple Users:
for user in alice bob charlie diana eve; do
  sudo useradd -m -g developers -s /bin/bash $user
  echo "password123" | sudo passwd --stdin $user
  sudo chage -d 0 -M 90 $user
done
Verify: getent passwd | grep -E "alice|bob|charlie|diana|eve"

File Permission Fix for Web Server

2

📋 Scenario

Your web application in /var/www/myapp is having permission issues. The web server (running as user "www-data") can't read static files, and developers can't upload new files. Fix the permissions properly.

Requirements:
  • Web server must read all files (directories and contents)
  • Developers group must read, write, and execute
  • New files should inherit proper permissions (setgid)
  • Prevent others from accessing
  • Logs directory should be writable only by www-data
chmod chown setgid permissions
Set Ownership:
Change owner and group: sudo chown -R www-data:developers /var/www/myapp
• -R: Recursive (all files and subdirectories)
• www-data: Owner (web server user)
• developers: Group
Verify: ls -la /var/www/myapp
Directory Permissions:
Set directories to 2775: sudo find /var/www/myapp -type d -exec chmod 2775 {} \;
• 2: setgid bit (inherit group)
• 7: Owner (rwx)
• 7: Group (rwx)
• 5: Others (r-x, read only)
File Permissions:
Set files to 664: sudo find /var/www/myapp -type f -exec chmod 664 {} \;
• 6: Owner (rw-)
• 6: Group (rw-)
• 4: Others (r--)
This allows web server to read, developers to write
Logs Directory Restriction:
sudo chown www-data:www-data /var/www/myapp/logs
sudo chmod 700 /var/www/myapp/logs
This ensures only www-data can write logs, developers can't access
Test with: sudo -u www-data touch /var/www/myapp/logs/test.log

Disk Space Cleanup Investigation

3

📋 Scenario

Your production server is running out of disk space (90% full). You need to find what's consuming space, identify old log files, and clean up safely without disrupting services.

Requirements:
  • Find top 10 largest directories
  • Identify log files older than 30 days
  • Find files larger than 100MB
  • Compress old logs instead of deleting
  • Clean up /tmp and /var/tmp safely
du df find gzip
Disk Usage Analysis:
Check filesystem usage: df -h
Check directory sizes: sudo du -h --max-depth=1 / | sort -hr | head -10
Check specific directory: sudo du -sh /var/*
Identify large directories: sudo du -ah /var | sort -rh | head -20
Locate Large Files:
Files larger than 100MB: sudo find / -type f -size +100M -exec ls -lh {} \;
Sort by size: sudo find / -type f -size +100M -exec ls -lh {} \; | awk '{print $5, $9}' | sort -rh
Exclude mounted drives: sudo find / -xdev -type f -size +100M
Find and Compress Logs:
Find logs older than 30 days: sudo find /var/log -type f -name "*.log" -mtime +30
Compress them: sudo find /var/log -type f -name "*.log" -mtime +30 -exec gzip {} \;
Verify compression: ls -lh /var/log/*.gz
Delete compressed logs older than 90 days: sudo find /var/log -name "*.gz" -mtime +90 -delete
Safe Temp Cleanup:
Check /tmp usage: du -sh /tmp
Files older than 7 days: sudo find /tmp -type f -mtime +7 -delete
Empty directories: sudo find /tmp -type d -empty -delete
Similar for /var/tmp: sudo find /var/tmp -type f -mtime +7 -delete
NOTE: Don't delete files less than 24 hours old (may be in use)

Process Management for High CPU Usage

4

📋 Scenario

The server CPU is at 95% usage. You need to identify which process is causing it, check its details, and manage it appropriately without killing critical services.

Requirements:
  • Identify top CPU-consuming processes
  • Check process details and command line
  • Find process tree and parent-child relationships
  • Lower priority for non-critical processes
  • Kill only if necessary (gracefully first)
top ps nice kill
Real-time Monitoring:
Interactive monitoring: top (press 'P' to sort by CPU)
Better alternative: htop (more user-friendly)
One-time snapshot: ps aux --sort=-%cpu | head -10
Watch specific user: top -u www-data
Get Process Information:
Full command line: ps aux | grep [process-name]
Process tree: pstree -p [PID]
Detailed info: ps -f -p [PID]
Process files: ls -l /proc/[PID]
Check open files: lsof -p [PID]
Nice and Renice:
Current priority: ps -o pid,ni,comm -p [PID]

Graceful Termination:
Try graceful shutdown first: kill -15 [PID] (SIGTERM)
Wait 10 seconds, then check if still running
Force kill if needed: kill -9 [PID] (SIGKILL)
Kill by name: pkill -15 process-name
Kill all instances: killall process-name

Cron Job Setup for Automated Backups

5

📋 Scenario

Set up automated daily backups of your application database. Backups should run at 2 AM, compress the output, and send email notifications on failure.

Requirements:
  • Daily backup at 2:00 AM
  • Backup location: /backup/db/
  • Filename format: db_backup_YYYY-MM-DD.sql.gz
  • Keep only last 7 days of backups
  • Email notification on errors
crontab mysqldump gzip automation
Backup Script:
Create /usr/local/bin/db-backup.sh:
#!/bin/bash
DATE=$(date +%Y-%m-%d)
BACKUP_DIR="/backup/db"
mysqldump -u root -p'password' dbname | gzip > $BACKUP_DIR/db_backup_$DATE.sql.gz
find $BACKUP_DIR -name "*.gz" -mtime +7 -delete
Make executable: chmod +x /usr/local/bin/db-backup.sh
Understanding Cron:
Format: minute hour day month weekday command
Daily at 2 AM: 0 2 * * *
• 0 = minute (0-59)
• 2 = hour (0-23)
• * = any day, month, weekday
Edit crontab: crontab -e
Crontab Entry:
Edit root crontab: sudo crontab -e
Add line: 0 2 * * * /usr/local/bin/db-backup.sh >> /var/log/db-backup.log 2>&1
• Redirects output to log file
• 2>&1 captures errors too
List current jobs: crontab -l
Error Notification:
Add to script top: MAILTO="admin@company.com"
Or in crontab before the job: MAILTO=admin@company.com
Test manually: sudo /usr/local/bin/db-backup.sh
Check logs: tail -f /var/log/db-backup.log
Cron log: grep CRON /var/log/syslog

PasswordAuthentication no

LVM Setup for Dynamic Storage

9

📋 Scenario

Your database needs 500GB storage but you only have 200GB available. Set up LVM across multiple disks to create a flexible, expandable volume that can grow as needed.

Requirements:
  • Create LVM volume group from 3 disks (/dev/sdb, /dev/sdc, /dev/sdd)
  • Create 200GB logical volume for database
  • Format with ext4 and mount at /data/mysql
  • Add to /etc/fstab for automatic mounting
  • Plan for expansion when new disks arrive
lvm pvcreate vgcreate lvcreate
Create Physical Volumes:
Install LVM: sudo apt install lvm2
Initialize disks: sudo pvcreate /dev/sdb /dev/sdc /dev/sdd
Verify: sudo pvdisplay or sudo pvs
Check disk status first: lsblk
Create Volume Group:
Create VG: sudo vgcreate data_vg /dev/sdb /dev/sdc /dev/sdd
Verify: sudo vgdisplay data_vg
Check available space: sudo vgs
To extend later: sudo vgextend data_vg /dev/sde
Create and Format LV:
Create 200GB LV: sudo lvcreate -L 200G -n mysql_lv data_vg
Format: sudo mkfs.ext4 /dev/data_vg/mysql_lv
Create mount point: sudo mkdir -p /data/mysql
Mount: sudo mount /dev/data_vg/mysql_lv /data/mysql
Verify: df -h | grep mysql
Auto-mount Configuration:
Get UUID: sudo blkid /dev/data_vg/mysql_lv
Add to /etc/fstab:
/dev/data_vg/mysql_lv /data/mysql ext4 defaults 0 2
Test: sudo mount -a
To resize later: sudo lvextend -L +100G /dev/data_vg/mysql_lv
Then: sudo resize2fs /dev/data_vg/mysql_lv

Systemd Service Creation for Custom Application

10

📋 Scenario

Your Node.js application needs to run as a systemd service. It should start on boot, restart on failure, and run as a specific user with proper logging.

Requirements:
  • Create systemd service unit file
  • Run as dedicated user "appuser"
  • Auto-restart on failure (max 3 times)
  • Start after network is available
  • Log to journald with proper rotation
systemd systemctl journalctl
Service Unit File:
Create /etc/systemd/system/myapp.service:
[Unit]
Description=My Node.js Application
After=network.target

[Service]
Type=simple
User=appuser
WorkingDirectory=/opt/myapp
ExecStart=/usr/bin/node /opt/myapp/server.js
Restart=on-failure
RestartSec=10
StartLimitBurst=3

[Install]
WantedBy=multi-user.target
Dedicated Service User:
Create system user: sudo useradd -r -s /bin/false appuser
• -r: System account
• -s /bin/false: No shell login
Set ownership: sudo chown -R appuser:appuser /opt/myapp
Verify: id appuser
Service Management:
Reload daemon: sudo systemctl daemon-reload
Start service: sudo systemctl start myapp
Check status: sudo systemctl status myapp
Enable on boot: sudo systemctl enable myapp
View logs: sudo journalctl -u myapp -f
Environment Configuration:
Add to service file:
Environment="NODE_ENV=production"
Environment="PORT=3000"
EnvironmentFile=/etc/myapp/env
Or create /etc/myapp/env with KEY=VALUE pairs
Secure sensitive data: sudo chmod 600 /etc/myapp/env
Test: sudo systemctl restart myapp

Kernel Parameter Tuning for Performance

12

📋 Scenario

Your high-traffic web server is experiencing connection timeouts and slow response times. Tune kernel parameters to handle more connections, optimize network stack, and improve overall performance.

Requirements:
  • Increase max open files limit to 65535
  • Optimize TCP settings for high connections
  • Enable TCP Fast Open
  • Adjust swappiness for better memory management
  • Make changes persistent across reboots
sysctl limits.conf tuning
Increase Open Files:
Current limit: ulimit -n
Edit /etc/security/limits.conf:
* soft nofile 65535
* hard nofile 65535
System-wide: Edit /etc/sysctl.conf
fs.file-max = 2097152
Apply: sudo sysctl -p
Network Stack Tuning:
Add to /etc/sysctl.conf:
net.core.somaxconn = 65535
net.ipv4.tcp_max_syn_backlog = 8192
net.ipv4.tcp_tw_reuse = 1
net.ipv4.tcp_fin_timeout = 15
net.ipv4.ip_local_port_range = 1024 65535
Apply immediately: sudo sysctl -p
Enable TFO:
Check current: cat /proc/sys/net/ipv4/tcp_fastopen
Enable client and server: echo 3 | sudo tee /proc/sys/net/ipv4/tcp_fastopen
Persistent in /etc/sysctl.conf:
net.ipv4.tcp_fastopen = 3
• 1 = client, 2 = server, 3 = both
Swappiness Adjustment:
Current value: cat /proc/sys/vm/swappiness
Set to 10 (less aggressive): sudo sysctl vm.swappiness=10
Permanent in /etc/sysctl.conf: vm.swappiness = 10
View all settings: sudo sysctl -a | grep -E 'tcp|file'
Verify after reboot: cat /proc/sys/vm/swappiness

🎯 Linux Best Practices

  • Security: Always use principle of least privilege, disable root SSH, use key-based auth
  • Updates: Keep system and packages updated, subscribe to security mailing lists
  • Backups: Implement 3-2-1 rule (3 copies, 2 media, 1 offsite), test restores monthly
  • Monitoring: Set up proper logging, monitoring, and alerting from day one
  • Documentation: Document all configurations, maintain runbooks for common tasks
  • Automation: Automate repetitive tasks, use configuration management tools
  • Resource Limits: Use ulimit, cgroups to prevent resource exhaustion
  • File Systems: Choose appropriate FS (ext4 general, XFS large files, Btrfs snapshots)
  • Kernel Parameters: Tune sysctl settings for your workload, document changes
  • Services: Disable unnecessary services, use systemd for service management
  • Network Security: Use firewall, configure SELinux/AppArmor, regular security audits
  • Performance: Profile before optimizing, use appropriate tools (perf, iotop, sar)

⚠️ Common Linux Pitfalls

  • Running everything as root instead of using sudo for specific tasks
  • Not testing commands before running them in production (especially rm, dd)
  • Forgetting to make firewall rules persistent across reboots
  • Not monitoring disk space and inodes, leading to full filesystem
  • Editing config files without backing them up first
  • Not understanding the difference between hard and soft links
  • Misconfiguring /etc/fstab causing boot failures
  • Not using screen/tmux for long-running remote tasks
  • Ignoring SELinux/AppArmor denials instead of fixing policies
  • Not properly handling log rotation, filling up disk
  • Copying files without preserving permissions and ownership
  • Not validating syntax before restarting critical services
